    Chap1 Introduction

    ROS = 通信机制 + 开发工具 + 应用功能 + 生态系统 核心概念： 1. 节点 Node 节点是具体执行程序的单元，可以认为是进程/独立运行的可执行文件。

    不同节点的编程语言可以不同

节点的名字必须唯一

ROS Master 是管理节点的控制中心。
1. 为节点提供注册和命名服务。
2. 跟踪话题记录/服务通信，辅助节点之间相互查找链接。
3. 提供参数服务器，作为全局参数的字典。

    1. 话题 Topic 话题仅是不同节点间信息传输的通道，之间通信由发布和订阅实现。

      需要注意这种通信是单向的（由发布者到订阅者，不知道订阅者是否收到），订阅者可以不唯一，并且是异步的。

      具体的传输形式是以消息(message)传输，即一种具有特殊的数据结构的数据包。这种数据结构定义是与编程语言无关的，由.msg文件定义，编译过程中可以生成相应的语言的代码。 3. 服务 Service 是一种同步的，双向的，有反馈的通信方法。

      采用Server/Client模型，由客户端发布请求，服务器完成应答后返回数据

      数据类型由.srv文件定义 4. 参数 Parameters 节点可以向ROS Master设置变量，适合用来存储静态，非二进制，不怎么改变的数据。 5. 文件体系 最底层是功能包Package 其中包含源码，配置文件和数据定义 再往上是元功能包和

      话题通信的编程实现

      1. 实现发布话题的流程：
      2. 初始化ROS节点init(argc,argv,"node_name")
      3. 注册节点句柄ROS::NodeHandle

      4. 向ROS Master注册话题发布节点，内容包括话题名和消息类型 inline ros::Publisher ros::NodeHandle::advertise&lt;geometry_msgs::Twist&gt;(const std::string &amp;topic, uint32_t queue_size, bool latch = false)

        参数：topic即话题名，queue_size即队列长度，因为话题通信是异步的；latch(optional)是标识是否存下最后一个发布的message，以便有新的订阅者接入时直接发送。 3. 创建消息数据 4. 按一定频率循环发布消息publisher.publish()

      Publisher类：用于发布话题，这里NodeHandle的advertise方法用于向ROS Master告知我要向这个topic发布消息了，并且设定了队列长度，之后再调用.publish()方法就可以实现发布消息。

      注意调用NodeHandle的advertise方法时，明显没有显式调用之前init的节点，是因为advertise时，NodeHandle仅向ROS Master告知现在系统内某个节点会向这个指定的topic发布消息，但是他也不知道是哪个节点会发布，也不需要知道。之后有节点想要知道消息，也就只要订阅那个topic，那些节点也不知道是哪个节点在发布消息，只知道这个话题里面会有消息出来。所以消息就是一个通道，两侧的节点都是双盲的。
      2. 订阅话题的流程 1. 定义一个回调函数，用于处理收到的消息 2. 初始化节点句柄 3. 订阅消息 inline ros::Subscriber ros::NodeHandle::subscribe<msg_type>;(const std::string &amp;topic, uint32_t queue_size, void (*fp)(msg::ConstPtr &amp;), const ros::TransportHints &amp;transport_hints = ros::TransportHints())

      参数：topic是想要订阅的话题 fp是回调函数，TransportHints()是什么函数？

      1. 循环等待话题消息，接收后进入回调函数，在回调函数中进行消息处理。
      2. 自定义消息类型
      3. 在.msg文件中定义，例如name.msg
      4. 在调用时包含头文件,例如name.h
      5. 之后就按正常的订阅和发布处理，不过消息的命名空间和类型都按自定义的来。

    服务通信的编程实现

    1. 实现client端
      1. ros::service::waitForService方法，等待某个服务有无进行
      2. ros::ServiceClient是一个服务客户端的类，由NodeHandle的serviceClient方法可以获得这个类的实例，serviceClient<msg_type>(string &service_name, bool persistent = false, ros::M_string &header_values = ros::M_string()) 参数：服务名，persistent先不管
      3. 初始化消息srv,注意消息是包含request和response的
      4. 用serviceClient的call方法可以发送request。
      5. 获得的response返回到srv.response中，调用这个就可以看到返回的结果。
    2. 实现server端
      1. 定义一个回调函数，处理server收到的request
      2. 初始化节点
      3. ros::ServiceServer是一个服务服务器端的类，由NodeHandle的advertiseService方法可以向ROS Master注册这个服务，ros::NodeHandle::advertiseService<...>(const std::string &service, bool (*srv_func) (srv::Request &, srv::Response &)) 参数：服务名，回调函数srv_func，注意格式
      4. 用spin()或者spinonce()可以从回调函数队列中获得带实际参数的回调函数，之后在回调函数内处理
    3. 实现自定义服务消息
      1. 在.srv文件中定义服务的消息类型，
      2. 加入编译所需的依赖和包，进行编译，得到头文件
      3. 在client和server的文件中调用头文件，即可使用
